# Anti-AI-slop checklist

Four layers, in priority order: hard rules (must follow), soft rules (strongly recommended), good-taste anchors (mental references), pre-flight scan (lint step). Beyond the hard rules, the two themes have their own conventions.

## Hard rules — must follow (apply to both themes)

### Rule 1: No purple/violet/indigo gradients anywhere

This is the #1 AI-generated UI tell. No `linear-gradient(... purple, indigo, violet, fuchsia, ...)` in any form — not as backgrounds, not as accents, not as hero sections. Solid colors only for backgrounds. Hairline accent bars only for accents.

Banned hex ranges (rough guide): anything `#5x-7x` in the `R<B` blue-purple zone, plus pure purples like `#8B5CF6`, `#A855F7`, indigo `#6366F1`, violet `#7C3AED`, magenta/fuchsia.

### Rule 2: No emoji as section icons or decorative bullets

Emoji 🚀 📊 ✨ 💡 ⚡ 🎯 ✅ ❌ ⚠️ etc. as section headers, status indicators, or decoration is the second-biggest AI tell. Use:

- **Status**: geometric Unicode shapes (●, ▸, ■, ○, ▲, ✓, ✗) in colored text, NOT emoji
- **Section markers**: a thin accent bar before the heading, NOT emoji
- **Bullets**: standard `<ul>` markers or none, NOT emoji
- **Decorative**: just don't

The allowed-shapes list is exactly: ● ▸ ■ ○ ▲ ✓ ✗ — pure geometric/check Unicode. These render in the document's text color (no platform-specific emoji rendering), survive copy-paste and print, and don't carry the "AI chatbot tone" of emoji like 🎯 or ✅. If you reach for any glyph in the U+1F300–U+1F9FF range or the emoji-presentation pile (✅ ❌ ⚠️ etc. — those look like check/cross/warning but render as colored emoji on most platforms), stop and use the geometric form instead. Note specifically: `✅` (U+2705 WHITE HEAVY CHECK MARK) is emoji; `✓` (U+2713 CHECK MARK) is geometric — they look similar but are different code points with different rendering. Use the latter.

Emoji that appears in source content (because the author wrote it) is fine — render it as written. The skill never *adds* emoji.

### Rule 3: No `Inter` (or `Roboto`, `Open Sans`, `Poppins`, `Lato`, `Nunito`, `system-ui`) as the primary body font

Body font must be either:
- A real serif (Charter, Source Serif, Iowan, etc.) — editorial theme primary
- A distinctive sans like `Söhne` or `IBM Plex Sans` — dashboard theme primary

`Inter` is allowed only as a fallback deep in a font-stack chain. Same for `system-ui` — fine as fallback, not primary.

## Soft rules — strongly recommended (other patterns to avoid, not absolute)

- **Glassmorphism** (`backdrop-filter: blur()`) — banned
- **Large drop shadows on content** (`box-shadow: 0 20px 40px ...`) — banned. Sticky nav can have a 1px border or a 4px subtle shadow, max.
- **Every block in a rounded box with a soft shadow** — looks like a dashboard mockup. Dashboard theme uses borders, not soft cards.
- **Fade-in-on-scroll animations** — banned. The document is paper, not a marketing site.
- **Hero sections** with centered large text on full-width colored background — banned.
- **"AI-generated by..." badges** — banned. Don't add attribution that wasn't requested.
- **Adding decorative content the source doesn't have** — banned. No "Let's dive in", no "Here's the magic", no inventing summary paragraphs.
- **Agent-shorthand leaking into the rendered HTML** — soft rule but important. The source MD is often written for engineers / AI agents and is dense with unexplained acronyms, copy-pasted issue titles as task names, and inline code stacked four-deep. If you render that directly, the HTML is paper-pretty but cognitively unreadable to a cross-functional audience. Smells to catch:
  - An H2 that drops straight into a list or table with no framing sentence
  - An acronym appearing for the first time without a plain-language hint
  - A task or list-item name > 60 chars or containing ≥ 3 inline `<code>` blocks, with no sub-line explanation
  - A paragraph with 4+ inline `<code>` blocks all at the same visual weight
  - ADR / decision bodies as one undifferentiated paragraph instead of a selection/why/cost skeleton
  - Risk callouts that don't visually separate "what goes wrong" from "mitigation"

  When you see these, apply the corresponding patterns from `references/readability.md`. The source isn't wrong — it's appropriate for source-of-truth. But the rendered HTML's job is to lift it for human readers.

- **Status counts that lie about scope** — soft rule but important. A status-glance panel showing `Blocked: 2` where the 2 items are out-of-scope follow-ups (e.g. "won't be in this release", "tracked but deferred") misleads a leader's 30-second scan. If a counter pulls in anything outside the active scope, either filter to the active scope only, or split the out-of-scope items into a separate panel — "Standing follow-up · not blocking [scope]" — using `.status-standing` (see `core-tokens.md`), not `.status-blocked`. Don't conflate "we're tracking this" with "this is blocking us". Same rule for `Pending`, `In progress`, and the percent-complete bar: each should count only what's inside the plan's own scope.

- **Language register mismatch within one document** — soft rule, often noticed last and most. A Chinese document with filter buttons labeled "可派 Agent / 需 Human" but row tags rendering as `AGENT` / `HITL` is using two different languages for the same concept in the same surface. Pick one register per concept and stay consistent across filter buttons, row tags, label maps, status badges, and any other element where that concept appears. The same applies in reverse — don't mix `Done` and `已完成` in adjacent table cells, or `Risk` and `风险` in adjacent callout labels. Match the document's primary language; if the source is Chinese, render concept-labels in Chinese throughout; if English, English throughout. Bilingual is acceptable only when the *audience* is explicitly bilingual and every concept appears as `English / 中文` consistently.

## Images — referential allowed, decorative banned

Tech documents sometimes legitimately need images: a screenshot of the current UI next to the proposed one, a photo of physical hardware in an IoT / Smart Building spec, a reference architecture sketch the team drew on a whiteboard, a vendor logo for attribution. The previous version of this skill banned "stock illustrations / hero images / generated graphics" but didn't carve out the legitimate cases, which left a gap. This section closes it.

**Allowed (referential)**:

- **Screenshots** of an existing or proposed UI, when the source MD references that UI specifically. Before/after UI comparisons especially benefit.
- **Photos of physical artifacts** — hardware in IoT specs, physical layouts in Smart Building docs, board prototypes. Real objects the doc is about.
- **Reference architecture diagrams** the team drew (whiteboard photos, drawing-app exports). When Mermaid + Pattern 9 SVG can't carry it and the source has an actual artifact, embed the artifact.
- **Vendor / product logos** when introducing a tool and the source uses the logo, at small size (≤ 32px tall, inline with the term). Don't make a logo wall.

**Banned (decorative or fabricated)**:

- **AI-generated illustrations** of any kind — avatars, decorative scenes, hero artwork, "abstract tech" backgrounds. The doc is paper.
- **Stock photography** as section dividers or atmosphere. Pixabay / Unsplash / generated-by-DALL-E all fall here.
- **Hero images** on full-width colored backgrounds at the top of the doc.
- **Decorative icons** beyond the geometric Unicode set from Rule 2. No image icons replacing emoji.
- **Photos / illustrations the source MD doesn't reference.** If the author didn't put it in the source, the skill doesn't invent it.

**How to embed** (single-file constraint):

- **Prefer data URLs** for images < 200 KB each, total image payload ideally < 2 MB. Keeps the HTML self-contained — opens from email attachments, intranet shares, offline.
- **External URLs** are acceptable for larger images, but the HTML degrades to a missing-image box when opened offline. Add a small note in the `<figcaption>` like "Image at <span class='url'>example.com/...</span> — won't display offline."
- **Always wrap in `<figure>` + `<figcaption>`.** Caption explains what the image shows; image alone has no semantic role.
- **Always provide `alt`** for the `<img>` — what the image conveys, not "screenshot". Screen readers and broken-image fallbacks both depend on it.
- **Size constraints in markup**: set explicit `width` / `max-width` on the `<img>`; let height stay auto. Never let an image push past `--content-max-wide`.

**Don't**:

- Don't convert a Mermaid diagram or Pattern 9 SVG into an image. Those mediums are queryable, scalable, and printable; images are not.
- Don't use images to bypass anti-slop rules. A gradient hero saved as a JPG is still a gradient hero.
- Don't embed images that contain text-only content (tables, code, lists) — text in an image is unsearchable, untranslatable, and inaccessible. Render as actual HTML.

## Good-taste anchors (mental references — what we're aiming at)

Mental references for both themes:

For **editorial theme**:
- The Economist's article layout (serif, narrow column, one red accent)
- Tufte CSS (serif, marginalia, generous whitespace)
- Stripe's docs (clear hierarchy, lots of whitespace, restrained)
- Kenya Hara / MUJI catalog (extreme whitespace, one warm accent)

For **dashboard theme**:
- Linear's project view (sans, restrained, monochrome with one accent for status)
- GitHub's project boards (functional density, semantic color for state, no decoration)
- Vercel's analytics dashboard (clean sans, semantic color, sparse)
- Notion's database views (information-dense but readable)

In both cases: if the design would look at home in either a financial annual report (editorial) or a serious internal developer tool (dashboard), it's right. If it would look at home on a Series A startup's marketing page, it's wrong.

## Pre-flight scan (lint step before output)

Before writing the final HTML, search the proposed `<style>` block for these strings and reject if found:

- `gradient` involving any purple/violet/indigo/fuchsia color
- `backdrop-filter`
- `box-shadow:` with a blur radius > 8px (allowed: small subtle shadow on sticky nav, max blur 8px)
- `Inter` as `font-family:` primary (only allowed as a deep fallback)
- `border-radius:` value > 8px on content blocks (chips/buttons OK at 4px, anything bigger smells)
- `content:` properties containing emoji codepoints (U+1F300–1F9FF range, or named emoji like 🚀)

Scan the HTML body for:

- Emoji characters in any heading, navigation, status indicator, or decoration position
- More than ONE element with `background: var(--accent)` (the accent is for hairlines and small details, not large fills)
- A `<section class="hero">` or similar centered hero block

If any of these fire, revise before output.
