Risks

Four risks, sorted by blast radius. The first is the load-bearing one — silent divergence eats the rest of the plan.

Risk · Silent data divergence under dual-write

If reconciliation does not catch a write that landed in v13 but not v16 (or vice versa), cutover will surface it as missing rows in production. The blast radius is unbounded until reconciliation runs again.

Mitigation: reconciliation runs every 15 minutes against row counts and partition checksums. Any non-zero diff pages the on-call engineer and pauses canary rollout. A zero diff sustained for 48 hours is the precondition for cutover sign-off (Task 3.1).

Risk · Query plan regressions on v16

The planner is not the same across major versions. Queries that were fast on v13 can be slow on v16 — typically by picking a different index or join order.

Mitigation: baseline plans captured in Phase 1, regressions diffed in Task 1.4 before canary. Any regression that survives into Phase 2 gets an index hint or a query rewrite before cutover.

Risk · Rollback complexity if cutover fails mid-window

Partial promotion can leave the cluster in a state where rolling back to v13 loses recent writes that already committed to v16.

Mitigation: rollback procedure rehearsed end-to-end in staging. Explicit no-go threshold at the 5-minute mark of the cutover window — past that, no rollback, only forward.

Risk · Extension behavior under production load

Extensions that pass smoke tests may behave differently under production traffic shape — different memory pressure, different lock contention.

Mitigation: Task 2.5 explicitly retests every installed extension under canary-load traffic before cutover sign-off.
