Dark mode for Glyph editor

A user-selectable dark theme for Glyph, the open-source code editor, applied consistently across editor surface, chrome, and embedded panels.

OwnerJ. Park StatusDraft v3 Updated2026-05-09 Target2026 Q3
TL;DR

Roughly 41% of Glyph users work in low-light environments, and the most upvoted public issue of the year is the dark-theme request. Existing third-party color extensions only re-skin the editor surface, producing a jarring two-tone window. V1 ships a unified dark theme by introducing a small CSS token vocabulary and overriding it at the document root — a third-party theme becomes a single declaration block instead of two.

Problem

Why we are doing this now, and what reading "dark mode" actually means for Glyph.

Glyph ships only a light theme. Roughly 41% of users work in low-light environments according to the latest opt-in environment survey, and the most upvoted issue in the public tracker over the last twelve months is the request for a built-in dark theme.

Third-party color extensions partially fill the gap, but they only re-skin the editor surface — chrome, settings panels, and the diff viewer stay light, which produces a jarring two-tone window that several long-form contributors have flagged as worse than no dark mode at all.

Critical decision

One choice anchors the rest of the design — call it out before the supporting decisions.

Load-bearing · Theme tokens, not theme stylesheets

Every color is referenced through a CSS custom property, and the dark theme is a single set of property overrides at :root[data-theme="dark"]. The alternative — shipping two parallel stylesheets — was rejected because it doubles maintenance and because third-party themes (which Glyph already supports) would have to be authored twice. With tokens, a third-party theme becomes a single declaration block instead of two.

Shape of the change

Anchor by contrast: what a component reads today, versus what it reads after the refactor.

Decisions

Three supporting decisions follow from the token choice — each gets a one-sentence selection, a one-sentence why, and a cost.

Use a token taxonomy, not raw colors Accepted · 2026-05-04

Selection: introduce a fixed set of named tokens (--surface, --surface-elevated, --ink, --ink-muted, --accent, --rule) and require every component to reference them. Disallow raw hex in component CSS.

Why: a token vocabulary makes the diff between light and dark a single 30-line block. It also gives third-party theme authors a stable contract — they override tokens, not internal class names.

Cost: refactoring existing components to consume tokens is a one-time cost (~1 week of cleanup). After that, any new color must be added to the taxonomy first.

Default to system preference, allow manual override Accepted · 2026-05-05

Selection: on first launch, read prefers-color-scheme and set the theme accordingly. The user's explicit choice in Settings overrides system preference and persists in editor config — not in the rendered document.

Why: matches the platform behavior most users already expect. Manual override is necessary because some users want dark mode on a light desktop (and vice versa) for ergonomic reasons.

Cost: introduces a small precedence rule (user choice beats system) that must be documented and tested. No runtime cost.

Defer syntax-highlighting palette to a separate effort Accepted · 2026-05-07

Selection: the dark theme uses the existing syntax palette unchanged for V1. A retuned syntax palette ships in a follow-up.

Why: syntax highlighting has its own contrast and accessibility concerns that warrant separate review. Bundling it into this PRD would expand scope and delay shipping the chrome-level dark mode users are actually asking for.

Cost: V1 will look "almost right but not quite" on heavily-syntax-highlighted files. A note in the release announcement sets the expectation.

Risks

Three risks, sorted by blast radius. The first is load-bearing — it can quietly undo the rest of the work.

Risk · Token coverage gaps

Any component that still uses a raw hex literal will render with a light surface against the dark chrome, reproducing the same two-tone effect this PRD is trying to eliminate. Gaps are easy to miss in review because they only manifest at runtime in dark mode.

Mitigation: a lint rule rejects raw color literals in component CSS, run in CI before merge. The rule has a tightly-scoped allowlist (test fixtures only) and fails the build on any other occurrence.

Risk · Contrast regressions

Color combinations that pass WCAG AA in light mode can fail in dark mode at the inverted lightness — text-on-surface contrast does not transfer symmetrically.

Mitigation: extend the visual-regression suite with a contrast check that walks every token pairing used in components. Release gates on AA pass.

Risk · Third-party theme breakage

Existing third-party themes that target internal class names rather than tokens will not adapt to the dark variant automatically.

Mitigation: publish a migration guide and keep the old class-name selectors working for one release cycle as a deprecation window before removing them.

Out of scope

Things we are explicitly not doing in V1 — each is tracked separately and may ship later.

  • Per-file or per-language theme overrides
  • A theme editor UI inside Glyph (users who want custom themes still hand-edit CSS files)
  • Retuned syntax-highlighting palette (see Decision 3)
  • High-contrast accessibility theme (separate accessibility track)
  • Automatic theme switching by time of day
